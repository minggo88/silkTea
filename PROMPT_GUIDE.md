# 프롬프트 회고 - 어떤 요청이 가장 정확하게 반영됐는가

오늘 작업 중 추가 수정 없이 한 번에 정확하게 반영된 프롬프트와, 여러 번 고쳐야 했던 프롬프트를 비교해서 패턴을 정리했습니다.

## 1. 한 번에 정확히 반영된 프롬프트

### (1) 수치/속성을 구체적으로 못박은 경우
> "Section 5: Market ... background: white, min-height: 100vh, padding: 100px 80px ... 상단 메인 이미지: height 480px, object-fit: cover ... 하단 이미지: grid-template-columns: 1fr 1fr, gap 16px ..."

> "Footer ... Background: #2C1A0E, padding: 60px 80px ... 사업자 정보(text-align center, font-size 12px, color rgba(255,255,255,0.35), line-height 2.0) ..."

→ CSS 속성값·색상코드·텍스트 내용까지 다 명시되어 있어서 해석의 여지가 없었고, 둘 다 한 번에 그대로 반영되어 추가 수정이 전혀 없었습니다. **가장 정확도가 높았던 패턴.**

### (2) "무엇과 같게/무엇 기준으로"처럼 비교 대상을 명시한 경우
> "끽다거 세션에서 모바일일때 글자크기가 위에 스토리 글자크기보다 작은데 스토리 글자크기와 같게 셋팅해줘"

→ "얼마로 줄여줘"가 아니라 "OO와 같게"라고 비교 기준을 줘서, 코드에서 정확한 숫자(17px)를 찾아 바로 맞출 수 있었습니다. 모호함이 전혀 없었던 케이스.

### (3) 원하는 "결과(동작)"를 설명한 경우 (메커니즘이 아니라)
> "company.jpg는 가로 기준으로 영역에 들어가게 해줘"

→ 이전에 "10% 작게 그려줘"로 시도했을 때는 실패했지만(아래 2-(1) 참고), "가로 기준으로 들어가게"라는 결과 중심 설명으로 바꾸자 `object-fit: cover` → `height: auto`로 근본 원인을 고쳐서 한 번에 해결됐습니다.

### (4) 구조를 모를 때 단서를 같이 준 경우
> "Find any border-top, hr, or decorative line element at the very top of the footer and delete it."

→ 정확한 클래스명을 몰라도 "border-top, hr, 또는 장식용 줄"이라는 후보를 짚어줘서 빠르게 정확한 요소(`.footer-divider`)를 찾아 삭제할 수 있었습니다.

### (5) 단순/명확한 단일 동작 요청
> "git에 올려줘", "버튼 사이 간격도 100씩 벌려줘", "footer에서 로고 지워줘"

→ 동작이 하나이고 모호함이 없어서 항상 한 번에 정확히 처리됐습니다.

## 2. 여러 번 고쳐야 했던 프롬프트

### (1) 증상 + "추측한 해결책"을 같이 준 경우
> "모바일에서 company.jpg가 파일이 잘려서 보이니까 모바일의 경우 현재보다 10% 작게 그려줘"

→ 실제 원인은 가로(width)가 잘리는 것이었는데, "10% 작게"라는 해결책을 먼저 제시해서 높이를 줄이는 잘못된 수정을 했습니다. 이후 "높이가 아니고 width가 잘려"로 정정이 필요했습니다. **증상만 말하고 원하는 결과(어떻게 보여야 하는지)를 설명했다면 한 번에 끝났을 작업.**

### (2) 퍼센트(%) 기준이 불명확한 경우
> "분류와 이미지의 height 를 80%정도 줄여줘"
> "왼쪽,오른쪽 페딩값을 100씩 더주고 그에 맞게 왼쪽 범위도 줄여줘"

→ "80% 줄이기"가 "현재의 20%로"인지 "현재의 80%로"인지, "그에 맞게"가 정확히 어떤 비율 조정을 의미하는지 등 해석이 여러 갈래였습니다. 매번 추론해서 진행했지만 의도와 다를 위험이 있었던 패턴.

### (3) 텍스트 블록을 붙여넣을 때 "이게 데스크탑/모바일 중 무엇의 변경인지" 명시하지 않은 경우
> (긴 본문 텍스트를 줄바꿈과 함께 붙여넣고) "모바일용 줄바꿈으로 만들었어 작업해줘"

→ 이미 데스크탑용 줄바꿈이 들어가 있는 문단에 모바일용 줄바꿈을 합치는 과정에서, 어떤 `<br>`이 데스크탑 전용이고 어떤 게 모바일 전용인지 헷갈려서 두 버전이 섞이는 실수가 있었습니다("두개가 섞였어"). 같은 문제가 다른 문단에서도 반복되어 총 세 번 정정했습니다.

### (4) 상대적인 크기 표현만 쓴 경우
> "버튼 가로로 좀더 크게 해줘"

→ 구체적인 기준이 없어 임의로 px 값을 정해야 했고, 이후 "width 100씩 추가해줘"처럼 다시 구체적인 수치 요청이 이어졌습니다.

### (5) 실험적으로 추가했다가 되돌린 경우
> "로고 아래에 #034c45 색으로 Silk Tealeaf Inc 써주는데..." → 잠시 후 "아니다 글자 넣기 전으로 돌려줘"

→ 결과를 보기 전에는 원하는 느낌인지 판단하기 어려운 디자인 실험이라 되돌림 자체는 자연스럽지만, "일단 시도해보고 별로면 되돌릴게" 식으로 미리 언급하면 의도 파악에 도움이 됩니다.

## 3. 앞으로 프롬프트 보낼 때 권장 방식

1. **숫자/색상/속성은 가능하면 그대로 명시**: "버튼 크게" 대신 "padding을 64px → 100px로" 처럼. 모호한 상대 표현(좀더, 약간)은 매번 임의 해석이 들어갑니다.
2. **퍼센트를 쓸 때는 기준을 명확히**: "80% 줄여줘"보다 "지금의 80% 크기로" 또는 "지금보다 20% 작게"처럼 기준점(현재값 대비 줄어든 결과 vs 줄어드는 양)을 구분해서 표현.
3. **증상보다 원하는 최종 모습을 설명**: "잘려서 안 보여 → 10% 작게" 대신 "가로 기준으로 전체가 다 보이게" 처럼 결과 상태를 설명하면 근본 원인(예: object-fit 속성)을 바로 고칠 수 있습니다.
4. **비교 대상이 있으면 그 대상을 지칭**: "OO 섹션의 글자 크기와 같게" 처럼 이미 존재하는 다른 요소를 기준점으로 주면 가장 정확합니다.
5. **데스크탑/모바일처럼 버전이 나뉘는 수정은 "이건 OO용이야"를 먼저 한 줄로 명시**: 특히 이미 한쪽 버전(데스크탑)에 줄바꿈/스타일이 들어가 있는 상태에서 다른 버전(모바일)을 추가할 때, "기존 데스크탑 버전은 그대로 두고 모바일에서만 적용" 같은 한 문장이 있으면 두 버전이 섞이는 실수를 막을 수 있습니다.
6. **구조를 모를 때는 후보 키워드를 같이 던지기**: "border-top, hr, 혹은 장식용 줄 같은 거 찾아서 지워줘"처럼 후보를 주면 탐색이 빨라집니다.
7. **실험적 변경은 "일단 보고 판단할게"라고 미리 알려주면** 어떤 의도로 시도하는 변경인지 더 잘 파악할 수 있습니다.
